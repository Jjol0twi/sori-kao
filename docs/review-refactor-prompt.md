# 재검토·리팩토링 목표 프롬프트

아래 프롬프트는 구현이 어느 정도 끝난 뒤, Codex에게 프로젝트를 다시 검토하고 필요한 경우 수정·재개발까지 맡길 때 사용한다.

```text
이 저장소는 한국어 단어/문장의 음운 특징을 분석해 카모지를 추천하는 Python 앱이다. 생성형 AI, LLM, 외부 API, 네트워크 호출 없이 동작해야 하며, 핵심 기준 문서는 docs/development-goal.md, docs/design.md, docs/build-prompt.md다. 커밋 규칙은 docs/git-conventions.md를 따른다.

목표:
현재 구현을 설계 의도와 대조해 재검토하고, 문서와 다르거나 사용자 의도와 다른 동작이 있으면 원인을 설명한 뒤 수정하라. 단순히 테스트를 통과시키는 것이 아니라, "음운 특징 기반의 설명 가능한 오프라인 추천기"라는 프로젝트 방향에 맞는지 계속 의심하면서 개선하라.

검토 기준:
1. 14차원 feature vector 순서가 코드, category_weights.json, 문서에서 모두 일치하는지 확인한다.
2. 모든 특징값이 [0, 1] 범위로 정규화되는지 확인한다.
3. auxiliary keyword/jamo bonus가 feature vector와 중복되지 않고 별도 점수 항으로만 처리되는지 확인한다.
4. category_scores = x @ W.T + bias + auxiliary_bonus 공식과 기여도 공식 x[i] * W[c][i]가 실제 코드와 맞는지 확인한다.
5. category_weights.json의 숫자는 논문 수치가 아니라 heuristic_initial임을 문서와 데이터 메타데이터가 일관되게 설명하는지 확인한다.
6. 10개 카테고리(응원, 기쁨, 당황, 분노, 긴장, 피곤, 장난, 사과, 감사, 집중)가 코드, JSON, 카탈로그, 테스트에서 빠짐없이 유지되는지 확인한다.
7. 같은 입력은 항상 같은 결과가 나와야 하므로 random, 네트워크, 외부 API 호출이 추천 경로에 없는지 확인한다.
8. 짧은 입력, 자모만 입력, 한글이 거의 없는 입력, 말줄임표, 상반된 신호에서 신뢰도/fallback이 과도하게 확신하지 않는지 확인한다.
9. 명시적인 감사/사과/응원/집중 키워드와 `semantic_hints` 의미 힌트가 보조 신호로만 쓰이되, 밝은/어두운 모음 점수에 완전히 묻히지는 않는지 확인한다.
10. README의 실행·테스트 명령이 깨끗한 환경에서도 실제로 동작하는지 확인한다.

작업 방식:
- 먼저 git status를 확인하고, 기존 사용자 변경을 되돌리지 않는다.
- 문서, 코드, 데이터, 테스트를 읽고 발견 사항을 우선순위로 정리한다.
- 버그나 의도 불일치를 발견하면 테스트를 먼저 추가하거나 기존 테스트를 강화한 뒤 수정한다.
- 리팩토링은 동작을 바꾸는 수정과 분리하고, 불필요한 대규모 구조 변경은 하지 않는다.
- 각 논리적 변경마다 테스트를 통과시킨 뒤 작은 단위로 커밋한다.
- 커밋 메시지는 Conventional Commits 형식과 docs/git-conventions.md를 따른다. AI/도구 공동작성 트레일러는 넣지 않는다.

필수 검증:
- uv run --with pytest python -m pytest
- 필요하면 주요 입력 예시(아싸아싸!!, ㅋㅋㅋㅋ 뭐야, 으으... 힘들다, 죄송합니다..., 고마워요ㅎㅎ, 빠샤 집중하자, 하암)를 직접 실행해 Top-3, 신뢰도, 추천 이유를 확인한다.

완료 보고:
수정한 파일, 발견한 문제, 왜 그렇게 고쳤는지, 실행한 테스트 결과, 생성한 커밋 목록을 간단히 정리하라. 아직 남은 리스크가 있으면 숨기지 말고 적어라.
```
