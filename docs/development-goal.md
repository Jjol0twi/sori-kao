# 개발 목표 문서: 한국어 음운 특징과 관습 표현을 활용한 규칙 기반 카모지 추천 앱

## 1. 프로젝트 한줄 목표

한국어 단어 또는 문장을 입력받아 초성, 중성, 종성, 반복 같은 음운 특징과 일부 관습 표현 보조 신호를 분석하고, NumPy 기반 점수 모델로 카모지 추천 카테고리 후보를 계산한 뒤 어울리는 기호 이모티콘/카모지를 추천하는 Python 앱을 만든다.

## 2. 프로젝트 배경

이 프로젝트는 사용자가 입력한 문장에 어울리는 기호 이모티콘을 추천하는 프로그램이다. 일반적인 감정 단어 사전 방식은 `행복`, `슬픔`, `화남` 같은 단어가 직접 포함되어 있을 때만 잘 작동한다. 하지만 실제 입력은 `아싸`, `으으`, `ㅋㅋㅋ`, `하...`, `빠샤`, `흐엉`, `두근두근`처럼 감정 단어가 없어도 소리와 형태에서 감정적 인상이 드러나는 경우가 많다.

따라서 이 앱은 단순 키워드 매칭보다 한국어 음운 상징과 의성어·의태어 연구를 참고한 규칙 기반 분석을 중심에 둔다. 입력 문장을 한글 음절 단위로 분해하고, 밝은/어두운 모음 경향, 된소리·거센소리, 받침, 반복 등을 수치화하여 카모지 추천 카테고리 점수를 계산한다. 다만 `하암`, `ㅋㅋ`, `ㅠㅠ`처럼 의미가 사용 관습에 크게 묶인 표현은 음운만으로 판정하지 않고 보조 신호로 분리한다. 자음/모음 교체 탐지는 연구적으로 중요하지만 알고리즘 난이도가 높으므로 MVP에서는 제외하고 향후 확장 기능으로 둔다.

중요한 점은 이 프로젝트가 생성형 AI나 대규모 학습 모델을 사용하지 않는다는 것이다. 또한 사용자의 실제 감정을 맞히는 감정 판정기가 아니라, 소리 인상과 보조 규칙을 합산해 카모지 후보를 정렬하는 설명 가능한 추천 시스템을 목표로 한다.

## 3. 카카오톡 이모티콘 추천 기능과의 차별점

최근 카카오톡은 사용자가 대화 입력창에 특정 단어를 입력하면 맥락에 맞는 이모티콘을 추천하는 기능을 제공한다. 공개 기사에 따르면 `감사합니다`, `ㅋㅋ`, `죄송`, `바빠` 같은 일상 표현이 입력되면 입력창 근처에 이모티콘이 표시되고, 보유한 이모티콘과 전체 이모티콘 중에서 추천이 이루어진다.

이 프로젝트는 카카오톡의 "입력 중 추천 UX"와 "맥락 기반 추천"이라는 아이디어는 참고하지만, 목표와 구현 방식은 다르게 잡는다.

| 구분 | 카카오톡 참고 사례 | 이 프로젝트 |
|---|---|---|
| 목적 | 실제 채팅 중 이모티콘 사용·구매 편의성 강화 | 음운 특징 기반 카모지 추천 원리 구현 |
| 추천 대상 | 사용자가 보유한 이모티콘, 전체 이모티콘 상품 | 텍스트 기반 기호 이모티콘/카모지 |
| 주요 근거 | 사용자 맥락, 보유/구매 가능 이모티콘, 서비스 데이터 | 한글 음운 특징, 규칙, NumPy 점수 |
| 개인화 | 사용자 이력과 상품 추천 가능 | 개인 이력 사용하지 않음 |
| AI 사용 | 서비스 내부 구현은 공개 범위 밖이며 AI 추천과 연결 가능 | 생성형 AI/API/대규모 모델 사용하지 않음 |
| 설명 가능성 | 사용자에게 내부 점수 이유가 자세히 노출되지는 않음 | 추천 이유를 규칙 단위로 표시 |

즉, 이 앱은 "카카오톡처럼 이모티콘을 잘 팔거나 개인화하는 앱"이 아니라, "한국어 입력의 소리 느낌을 분석해 설명 가능한 방식으로 카모지를 추천하는 학습용 Python 앱"이다.

이 차별점이 설득력을 가지려면 발표에서 "카카오톡보다 더 좋은 추천"이라고 주장하지 않아야 한다. 이 프로젝트의 강점은 서비스 추천 품질이 아니라, 추천 과정을 작게 분해하고 설명할 수 있다는 점이다. 따라서 비교 기준은 실사용 규모나 개인화 성능이 아니라 `규칙의 투명성`, `오프라인 재현성`, `음운 특징 실험 가능성`으로 둔다.

## 4. Kaggle 비지도 학습 사례 참고 범위

Kaggle의 unsupervised learning 사례는 모델을 그대로 가져오기 위한 자료가 아니다. 이 프로젝트는 대규모 데이터셋으로 군집 모델을 학습하는 방향이 아니라, 입력을 특징 벡터로 바꾸고 그 벡터를 기준으로 유사도와 점수를 계산하는 아이디어만 참고한다.

참고할 부분은 다음과 같다.

- 텍스트 입력을 여러 숫자 특징으로 바꾸는 feature vector 관점
- 각 감정·상황 카테고리를 하나의 기준 벡터로 두고 입력 벡터와 비교하는 방식
- 점수 차이를 기준으로 Top-K 추천 결과를 정렬하는 방식
- 나중에 데이터가 생겼을 때 군집화나 유사도 기반 개선으로 확장할 수 있다는 구조

MVP에서는 scikit-learn 기반 군집 학습보다 NumPy 배열 계산을 우선한다. 이유는 대학 과제 수준에서 구현과 설명이 쉽고, "왜 이 카모지가 추천되었는지"를 규칙 단위로 보여주기 좋기 때문이다.

## 5. 한국어 음운 상징 연구를 규칙으로 바꾸는 방법

한국어 의성어·의태어 연구에서는 반복, 부분 반복, 음운 교체, 모음조화, 음성 상징이 의미와 어감 차이에 관여한다고 본다[국립국어원, 1993]. 국립국어원 자료에서도 의성·의태어의 반복 가능성, 음절 반복, 음소 교체, 모음조화에 의한 어휘 대립을 다룬다. 또한 한국어 ideophone 연구에서는 밝은 모음과 어두운 모음의 대비가 작음/가벼움/긍정적 사건, 큼/무거움/부정적 사건 같은 인상과 연결될 수 있음을 설명한다[성제현, 2018]. 어말 자음 교체와 종성의 기능은 흉내말의 어말 자음 교체 연구를 참고한다[박동근, 2014].

다만 연구에서도 음운과 의미의 관계가 항상 절대적이라고 보지는 않는다. 예를 들어 밝은 모음과 어두운 모음의 의미 대응은 경향적이며, 실제 말뭉치에서는 예외가 존재한다. 따라서 이 앱은 "이 모음이면 반드시 이 감정"이라고 단정하지 않고, 여러 특징을 합산하는 점수 모델로 구현한다.

프로그램 규칙으로 바꿀 수 있는 요소는 다음과 같다.

| 연구/관찰 요소 | 프로그램 특징으로 바꾸는 방법 | 추천에 쓰는 방식 |
|---|---|---|
| 밝은/어두운 모음 대비 | 중성을 양성, 음성, 중성 계열로 분류 | 기쁨, 장난, 긴장, 피곤 등 카테고리 점수에 반영 |
| 반복 | 같은 음절, 같은 자모, 같은 패턴 반복 탐지 | 강조, 지속, 장난, 응원, 긴장 점수에 반영 |
| 자음 교체 | 비슷한 위치의 종성·초성 변화 탐지 | 변화감, 흔들림, 장난스러움 보조 점수 |
| 모음 교체 | 같은 자음 틀에서 중성만 바뀌는 패턴 탐지 | 크기감, 밝기감, 정서 강도 보조 점수 |
| 된소리/거센소리 | ㄲ, ㄸ, ㅃ, ㅆ, ㅉ, ㅋ, ㅌ, ㅍ, ㅊ 비율 계산 | 강함, 분노, 응원, 놀람 점수 |
| 비음/유음 | 초성 기준 ㄴ, ㅁ, ㄹ 비율 계산(초성 ㅇ은 무음으로 보아 제외) | 부드러움, 피곤함, 울먹임, 안정감 점수 |
| 종성 ㅇ/ㅅ 등 | 받침 종류 집계 | 울림, 미완, 여운 같은 설명 생성 |
| 문장부호 | `!`, `?`, `...`, `ㅋㅋ`, `ㅠㅠ` 등 탐지 | 보조 신호로만 사용 |

위 표의 `변화감`, `흔들림`, `크기감`, `밝기감`, `울림`, `여운` 같은 표현은 공식 추천 카테고리가 아니라 내부 보조 점수나 설명 생성을 위한 해석 용어다. 공식 추천 카테고리는 §7의 10개 카테고리로 제한한다.

### 문헌 근거와 개발자 설정값의 구분

이 프로젝트에서 문헌은 "어떤 음운 특징을 볼 것인가"를 정하는 근거로 사용한다. 반면 `category_weights.json`에 들어갈 실제 숫자 가중치, `bias`, `auxiliary_bonus` 상한, 신뢰도 임계값, Top-3 평가 기준은 문헌에서 직접 제시된 값이 아니라 MVP 구현을 위해 정하는 초기 휴리스틱이다.

따라서 발표나 보고서에서는 "논문에서 반복 가중치를 +1.0으로 제시했다"처럼 말하지 않는다. 정확한 표현은 "문헌에서 반복, 교체, 모음조화, 종성 교체가 어감 차이에 관여할 수 있음을 확인했고, 이 특징들을 프로그램에서 점수화하기 위해 초기 가중치를 사람이 설정했다"이다.

| 구분 | 문서에서 말할 수 있는 내용 | 주의할 점 |
|---|---|---|
| 문헌 기반 요소 | 반복, 부분 반복, 자음 교체, 모음 교체, 모음조화, 밝은/어두운 모음 대비, 종성 교체를 특징 후보로 삼는다 | 감정 정답을 보장한다고 말하지 않는다 |
| 구현 편의 추가 요소 | 종성 카운트, 문장부호, 자모 반복, 보조 키워드 힌트처럼 실제 앱 동작을 안정화하기 위해 넣는 특징이 있을 수 있다 | 모든 특징이 논문에서 직접 나온 것처럼 말하지 않는다 |
| 개발자 설정 요소 | feature 정규화 방식, 실제 가중치 숫자, bias, auxiliary bonus, confidence threshold, Top-3 평가 기준 | 논문에서 나온 수치처럼 표현하지 않는다 |
| 해석 방식 | 추천 결과는 문헌 기반 특징을 활용한 규칙 기반 점수화 결과다 | 사용자의 실제 감정을 판정한다고 주장하지 않는다 |

## 6. 입력 처리 흐름

프로그램의 기본 흐름은 다음과 같다.

```text
문장 입력
→ 한글 음절 필터링 및 보조 기호 탐지
→ 초성·중성·종성 분해
→ 음운 특징 추출
→ NumPy feature vector 생성
→ 카테고리별 가중치 행렬과 점수 계산
→ 보조 신호 탐지(키워드·자모·의미 힌트)
→ 카모지 추천 카테고리 Top-K 산출
→ 카테고리에 연결된 카모지 추천
→ 추천 이유 설명 출력
```

예를 들어 `아싸아싸!!`가 입력되면 반복, 밝은 모음, 느낌표, 강한 리듬이 감지될 수 있다. 이 경우 `응원`, `기쁨`, `장난` 카테고리 점수가 올라가고 `(ง •̀ω•́)ง`, `٩( ᐛ )و`, `ヽ(>∀<☆)ノ` 같은 카모지를 추천할 수 있다.

반대로 `으으... 힘들다`가 입력되면 어두운 모음, 반복, 말줄임표, 피로 관련 보조 단어가 감지될 수 있다. 이 경우 `피곤`, `긴장`, `당황` 카테고리 점수가 올라가고 `(›´-‹ )`, `(＿ ＿*)`, `(；￣Д￣)` 같은 카모지를 추천할 수 있다.

## 7. 추천 카테고리 예시

MVP에서는 너무 많은 감정을 다루지 않고, 과제에서 설명하기 쉬운 감정·상황 카테고리 10개를 우선 사용한다.

| 카테고리 | 설명 | 카모지 예시 |
|---|---|---|
| 응원 | 힘을 주거나 격려하는 상황 | `(ง •̀ω•́)ง`, `٩( 'ω' )و`, `ᕦ(ò_óˇ)ᕤ` |
| 기쁨 | 밝고 즐거운 반응 | `ヽ(>∀<☆)ノ`, `(*^▽^*)`, `٩( ᐛ )و` |
| 당황 | 놀람, 어색함, 예상 밖 상황 | `(・・;)`, `(⊙_⊙;)`, `Σ(°ロ°)` |
| 분노 | 강한 불쾌감이나 화남 | `(ง •̀_•́)ง`, `٩(╬ʘ益ʘ╬)۶`, `ಠ_ಠ` |
| 긴장 | 초조함, 불안, 압박감 | `(；ﾟДﾟ)`, `(￣□￣;)`, `(汗)` |
| 피곤 | 지침, 무기력, 힘 빠짐 | `(＿ ＿*)`, `(´-ω-)`, `(×﹏×)` |
| 장난 | 웃김, 놀림, 가벼운 농담 | `( ͡° ͜ʖ ͡°)`, `(^_<)`, `ヽ(°〇°)ﾉ` |
| 사과 | 미안함, 조심스러운 태도 | `m(_ _)m`, `(シ_ _)シ`, `(｡•́︿•̀｡)` |
| 감사 | 고마움, 정중한 반응 | `(*ᴗ͈ˬᴗ͈)ꕤ*.ﾟ`, `(人´∀｀)`, `(*_ _)人` |
| 집중 | 몰입, 결심, 작업 모드 | `( •̀ .̫ •́ )✧`, `(ง •̀_•́)ง`, `(￣^￣)ゞ` |

카모지는 현재 `data/kaomoji_catalog.json`에 분리해 관리한다. 카테고리별 후보와 크기 태그를 데이터로 두어, 추천 로직이 카모지 문자열을 코드에 직접 박아 넣지 않게 한다.

## 8. 추천 이유 설명 방식

추천 결과는 카모지만 보여주지 않고, 왜 그 결과가 나왔는지도 함께 설명한다. 예시는 다음과 같다.

```text
입력: 아싸아싸!!
추천: (ง •̀ω•́)ง
카테고리: 응원
이유:
- 같은 음절 패턴이 반복되어 강한 리듬감이 감지되었습니다.
- 밝은 모음 계열이 많아 긍정적·가벼운 인상 점수가 올라갔습니다.
- 느낌표가 있어 에너지 점수가 보정되었습니다.
```

설명은 학술적으로 단정하지 않고, "감지되었습니다", "점수가 올라갔습니다", "보조 신호로 반영되었습니다"처럼 점수 모델의 결과로 표현한다.

추천 이유는 실제 감정을 맞혔다는 증명이 아니라, 프로그램이 어떤 규칙으로 점수를 계산했는지 보여주는 설명이다. 예를 들어 `밝은 모음이 많아서 기쁨입니다`가 아니라 `밝은 모음 비율이 높아 기쁨 카테고리 점수에 일부 반영되었습니다`처럼 표현해야 한다.

## 9. MVP 범위

MVP에서 구현할 기능은 다음과 같다.

- tkinter 기반 간단한 GUI에서 한국어 단어/문장 입력 (작은 입력창이 입력 후 아래로 확장되며 결과 표시)
- 한글 음절의 초성·중성·종성 분해
- 반복, 모음 계열, 자음 계열, 받침, 문장부호 특징 추출
- NumPy 기반 feature vector 생성
- 카테고리별 가중치 행렬로 점수 계산
- Top-3 카테고리와 카모지 추천
- 추천 이유 출력
- 규칙과 카모지 목록을 별도 데이터 파일로 관리

MVP에서 제외할 기능은 다음과 같다.

- 생성형 AI, OpenAI API, LLM 사용
- 사용자별 구매 이력 또는 사용 이력 기반 개인화
- 실제 카카오톡 이모티콘 상품 추천
- 대규모 학습 데이터 수집
- 딥러닝 모델 학습
- 자음/모음 교체 자동 탐지
- 모바일 앱 배포
- 카카오톡 UI 복제
- 추천 결과의 심리학적 정확성 보장

카모지 데이터는 외부 API를 실시간 호출하지 않고, 프로젝트 내부의 `data/kaomoji_catalog.json`에 직접 정리한다. kaomojis.jp API, GitHub의 kaomoji dataset, Hugging Face의 kaomoji caption dataset 같은 공개 자료는 후보 수집과 카테고리 참고용으로만 사용한다. 이렇게 하면 발표 환경에서 인터넷이 없어도 결과를 재현할 수 있고, 데이터 출처와 카테고리 매핑을 직접 설명할 수 있다.

## 12. 기획서 비판적 검토 요약

현재 기획서의 강점은 다음과 같다.

- 카카오톡의 이모티콘 추천 기능을 참고하되, 상업적 개인화 추천이 아니라 설명 가능한 음운 기반 추천으로 방향을 분리했다.
- 생성형 AI/API를 쓰지 않고 NumPy 점수 모델로 구현하겠다는 제약이 분명하다.
- 입력 처리 흐름이 `한글 분해 → 특징 추출 → 점수화 → 카모지 추천 → 이유 설명`으로 나뉘어 있어 구현 단위를 나누기 쉽다.
- 추천 결과를 단순 출력하지 않고 추천 이유까지 보여주므로 발표에서 내부 원리를 설명하기 좋다.

현재 기획서의 약점과 보완 필요점은 다음과 같다.

- 음운 상징과 감정 사이의 연결이 경향인지 규칙인지 더 조심스럽게 구분해야 한다.
- 밝은 모음, 어두운 모음, 된소리, 반복이 항상 같은 감정을 만들지는 않으므로 반례 처리가 필요하다.
- `ㅋㅋ`, `ㅠㅠ`, `감사`, `미안` 같은 표현이 들어오면 키워드 매칭이 결과를 지배할 수 있으므로 보조 신호의 비중을 제한해야 한다.
- 카모지 데이터 출처를 외부 API로 둘지 로컬 DB로 둘지 명확히 해야 한다.
- 추천 성능을 어떻게 평가할지, 어떤 경우를 실패로 볼지 기준을 더 구체화해야 한다.
- 입력이 너무 짧거나 한글이 거의 없는 경우의 fallback 정책이 필요하다.

## 13. 반례/실패 가능성

이 프로젝트는 음운 특징과 감정 인상의 관계를 실험적으로 활용하지만, 그 관계가 항상 맞는 것은 아니다. 따라서 다음 반례와 실패 가능성을 문서와 구현에서 인정해야 한다.

| 반례/실패 가능성 | 예시 | 보완 방향 |
|---|---|---|
| 밝은 모음이 있어도 긍정 감정이 아닐 수 있음 | `화나`, `피곤하다`, `아프다`, `배고파`, `아... 망했다` | 밝은 모음 가중치를 낮추고, 명시적 반응·불편 표현은 의미 힌트와 낮은 신뢰도로 분리한다 |
| 관습적 의성어가 음운과 다르게 해석될 수 있음 | `하암`은 밝은 ㅏ 모음이라 `하하`와 가깝지만 관습적으로 하품을 뜻함 | `yawn` 의미 힌트로 처리하되 낮은 신뢰도로 표시한다 |
| 짧은 관습 반응·요청이 음운과 다르게 해석될 수 있음 | `이런`, `놀랐잖아`, `망할`, `화나`, `가지마` | `surprise_reaction`, `negative_reaction`, `pleading_request` 같은 의미 힌트로 처리하고 음운 기반 판정과 구분한다 |
| 명시적 고통·피로·불편 표현이 밝은 음운을 포함할 수 있음 | `맘이 아프다`, `피곤하다`, `배고파`, `힘들어` | `distress` 의미 힌트로 피곤 후보를 올리되 음운 기반 판정과 구분한다 |
| 어두운 모음이 있어도 부정 감정이 아닐 수 있음 | `우와`, `오오` | 반복과 감탄 패턴으로 기쁨/놀람 점수를 보정한다 |
| 된소리·거센소리가 항상 분노는 아님 | `빠샤`, `ㅋㅋㅋ`, `파이팅` | 응원/장난 카테고리와도 경쟁하게 한다 |
| 반복이 특정 감정을 보장하지 않음 | `구르구르`, `으으`, `ㅠㅠㅠ`, `아아...` | 반복 자체는 강도·리듬 점수로 두고, 자모만 반복된 입력은 일반 반복 점수가 아니라 자모 보조 패턴으로 처리한다 |
| 자모 웃음 표현이 완성형 음운 정보 없이 들어올 수 있음 | `ㄹㅇㅋㅋ`, `ㅋㅋ` | `ㅋ` 반복은 장난 보조 신호로 우선 처리하고 낮은 신뢰도로 표시한다 |
| 키워드가 음운 점수를 압도할 수 있음 | `감사합니다`, `죄송합니다` | 키워드는 auxiliary bonus로 제한하고 최종 이유에 보조 신호로 표시한다 |
| 카모지 문화권 차이 | 한국어 입력을 일본식 카모지로 추천 | 카테고리별 카모지를 직접 선별하고 과도하게 낯선 표현은 줄인다 |
| 짧은 입력의 정보 부족 | `아`, `음`, `ㅋ` | 신뢰도 낮음 표시 또는 기본 후보 추천 |
| 한글 외 입력 | `lol!!`, `123`, `ㅠㅠ`만 입력 | 한글 분석 실패 시 기호 기반 fallback 사용 |

발표에서는 이 한계를 약점으로 숨기지 않고, "규칙 기반 모델이므로 예외가 있고, 예외를 점수 조합과 fallback으로 다룬다"라고 설명한다.

## 14. 추천 평가 기준

추천 품질은 하나의 정답 카모지를 맞히는 방식으로 평가하지 않는다. 카모지는 취향과 문화 차이가 크기 때문에, MVP에서는 다음 기준을 사용한다.

- 기대 카테고리가 Top-3 안에 들어오면 성공으로 본다.
- 추천 이유가 실제 계산된 특징과 일치해야 한다.
- 음운 특징 없이 키워드만으로 결과가 결정된 경우는 부분 성공으로 본다.
- 입력이 짧거나 모호한 경우에는 높은 확신 대신 fallback 또는 낮은 신뢰도를 표시한다.
- 최소 20개 이상의 테스트 문장을 만들어 카테고리별 편향을 확인한다.
- 기대 카테고리 라벨은 가중치 조정 전에 먼저 정하고, 일부 테스트 문장은 가중치 조정에 사용하지 않는 hold-out 사례로 남긴다. 이렇게 해야 개발자가 원하는 결과에 맞춰 가중치를 끼워 맞추는 자기충족 평가를 줄일 수 있다.

이 기준은 모델의 심리학적 정확성을 증명하기 위한 것이 아니라, 과제 수준에서 구현 결과가 설계 의도와 일치하는지 확인하기 위한 실용적 기준이다.
